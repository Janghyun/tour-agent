## 1. 요약

프로토타입은 브리프의 골격(채팅+접이식 패널, 카드 4종, 명시적 봇 호출, 방장 전용 확정, 단일 작업 일정, 흐름 A/B/C/E/F/D, 로비·방 만들기·초대 입장)을 충실히 구현했고, 브리프 범위를 넘는 보강(외부 링크 후보 추가, 후보 상세 팝업, 방 진입 화면)까지 추가했다. 백엔드 계약(present_* 카드, 명시적 호출, 단일 작업 일정, 방장 확정)과의 정합성은 대체로 좋고, "경쟁 안 동시 표시"나 "봇 자동 응답" 같은 금지 패턴 위반은 없다.

가장 큰 약점은 **모바일**이다. 브리프가 "모바일 가독성 1순위"로 못박았는데, 디자이너는 의도적으로 "데스크톱 먼저"로 진행했고(chat1.md), 그 결과 모바일 레이아웃이 사실상 부재하다. 미디어쿼리는 단 1개(`max-width:1080px`에서 패널 폭만 축소)뿐이며, 브리프가 명세한 모바일 핵심 요소(상단 요약 바, 하단 탭/바텀시트, 패널 기본 접힘)가 구현되지 않았다. 그 다음으로 색상 대비(접근성)와, 실제 백엔드 데이터(Kakao 좌표·영업시간 부재, present_* 비동기 스트리밍)를 붙였을 때 깨질 지점들이 구현 전에 해소되어야 한다.

## 3. 백엔드 계약 정합성

대부분 정합하며, 금지 패턴(봇 자동 응답, A/B/C 경쟁 안 동시 표시, 봇 임의 후보 추가)은 위반하지 않는다. 다만 구현 시 맞춰야 할 간극이 있다.

- **정합(양호)**
  - present_* 카드 계약: 카드 종류(places/itinerary/compare/map)가 TASK 6절의 `present_place_options`/`present_itinerary`/`present_map`과 1:1 대응. 프론트는 카드 페이로드를 받아 렌더하는 구조라 "텍스트 추측 파싱 안 함" 원칙과 맞다.
  - 명시적 호출만: 게이트 A(응답 여부) 개념이 UI에 반영됨(봇 부르기/슬래시 없이는 callsBot=false).
  - 단일 작업 일정 + 확정 스냅샷: 모델·문구 모두 계약대로.
  - 후보 등록 경로: 봇 임의 추가 없음. 옵션 카드 "추가" + (브리프 외) 사용자 명시 링크 추가. 둘 다 사용자 액션이라 제약 위배 아님.

- **맞춰야 할 간극(구현 전 확인)**
  1. **의도 분류가 프론트에 있음.** `data.jsx`의 `classifyIntent`(정규식)와 `Scenario.*`(하드코딩 페이로드)는 프로토타입용. 실제로는 게이트 B(Haiku)와 서브에이전트가 백엔드에서 판단하고, 프론트는 카드 페이로드만 받는다. 프론트 의도 분류는 폐기 대상임을 구현팀에 명확히 전달해야 한다.
  2. **옵션 비교는 "사용자 요청 시에만"인데 트리거가 약함.** 브리프·계약 모두 자동 제시 금지. 현재 `/검색` 등으로만 닿고, compare로 가는 명시 슬래시(예: `/비교`)가 슬래시 세트에 없다. 계약을 어기진 않지만 사용자가 비교를 부를 길이 불명확.
  3. **숙소 "지정" 후 실제 박별 동선 재계산 흐름이 UI에 없음.** 백엔드는 박별 숙소 기준으로 동선을 다시 계산(Kakao Mobility→NN+2-opt). 현재는 "숙소로 지정했어요" 버블만 뜨고 타임라인/지도가 갱신되지 않는다. 박마다 다른 숙소를 고르는 UI(슬롯별 변경)도 없다.
  4. **확정 후 작업본·확정본 분기 표현.** 계약상 확정 후에도 작업본은 계속 바뀐다. 현재 일정 탭은 작업본 하나만 보여주고 "확정본을 따로 보기"가 없다. 확정 스냅샷과 작업본을 나란히/전환해 보는 뷰가 빠졌다(4절 재검토).
  5. **영업시간 "확인 필요" 톤.** 계약은 Kakao가 영업시간 미제공 → WebSearch 교차확인 → 단정 대신 "확인 필요". 현재 카드/상세에는 영업시간 필드도, "확인 필요" 단서 표기도 없다(4절 데이터 갭과 직결).
  6. **큐잉/취소 동작의 단순화.** 현재 디바운스(1~2초 묶음), 다중 큐 표시, 취소 시 큐 비움 정책 등이 단순. 계약(논블로킹+큐)과 큰 충돌은 없으나, "취소는 명시 버튼만, 선점 없음"은 잘 지켰다.

결론: **계약 위반 없음.** 단 프론트의 인텔리전스(의도분류·시나리오 페이로드)는 데모용이며 실제는 백엔드 present_* 호출로 대체된다는 점, 그리고 숙소 재계산·확정본 뷰·영업시간 톤이 미구현이라는 점을 인계해야 한다.

## 4. UX 이슈 (심각도별)

### High

- **[H1] 모바일 레이아웃 부재 — 브리프 1순위 위반.**
  브리프는 모바일 가독성 1순위, 상단 요약 바(필수)·하단 탭(바텀시트)·패널 모바일 기본 접힘을 명세. 그러나 미디어쿼리는 `max-width:1080px`에서 패널 폭만 340px로 줄이는 1줄이 전부다. 좁은 폭에서 채팅+패널이 가로로 함께 눌려 둘 다 못 쓰게 된다. 데스크톱 우선 결정(chat1.md)은 이해되나, 브리프의 최우선 요구와 정면 충돌하므로 **구현 전 사용자 합의가 필요**한 사안이다.

- **[H2] 색상 대비 접근성 미달(WCAG AA).**
  크림 배경(#F6F1E8) 위에서 `--ink-3`(#938B7C, 메타·서브텍스트로 광범위 사용) 대비 3.00 → 본문 기준(4.5) 미달. `--ink-4`(#B6AE9E, placeholder·faint) 1.96 → 큰 글자에서도 미달. 흰 배경 위 `--ink-3`도 3.37로 본문 미달. 액센트 그린 버튼의 흰 글자(#FFF on #1F8A5B)는 4.33으로 일반 텍스트 기준 미달(큰 글자만 통과). 거리·평점·시간 등 핵심 메타가 ink-3/ink-4를 쓰므로 저시력·야외(여행 도메인) 사용에서 가독성 문제가 크다.

- **[H3] 에러 상태 전무.**
  링크 파싱 실패(잘못된 URL·미지원 도메인), 검색 0건, 일정 생성 실패/타임아웃, 네트워크/WebSocket 끊김, 큐 작업 실패 등 어떤 실패 경로도 화면이 없다. 실제 서비스에서 가장 자주 마주칠 상태인데 디자인이 비어 있다.

### Med

- **[M1] 시나리오 하드코딩으로 인한 인터랙션 단절감.**
  `modify`는 "우도"만, compare 선택·숙소 지정은 버블 안내만 하고 실제 타임라인/지도가 바뀌지 않는다. 프로토타입 한계지만, 이 "말은 하는데 패널은 안 바뀐다"는 패턴이 실제 구현에서도 남으면 신뢰를 깬다. 카드 액션과 패널 상태의 단일 출처 연결을 구현 시 필수화해야 한다.

- **[M2] 키보드 접근성/포커스 표시 부족.**
  대부분의 인터랙티브 요소가 `<button>`이라 기본 포커스는 가능하나, `:focus-visible` 스타일 정의가 없어 키보드 사용자가 현재 위치를 알기 어렵다(입력란만 focus ring). 클릭 가능한 후보 썸네일/이름은 `<div onClick>`이라 탭 포커스·엔터 동작 불가. 모달은 ESC 닫기·포커스 트랩·열릴 때 포커스 이동이 없다.

- **[M3] 지도 스크린리더/대체텍스트 공백.**
  지도 SVG는 `aria-hidden`, 핀은 텍스트 라벨 없는 아이콘 버튼이라 스크린리더로 동선/장소를 전혀 알 수 없다. 카테고리도 이모지+색으로만 구분(색맹·SR 취약). 카테고리 라벨 텍스트 병기, 핀 aria-label, 지도 대안 목록(일정 탭 타임라인)이 그 역할을 한다는 점 명시 필요.

- **[M4] 상단 요약 바 "펼침" 동작 누락.**
  브리프: 요약 바를 누르면 확정 일정이 펼쳐져야 함. 현재 topbar는 정적 텍스트라 모바일에서 확정 일정에 빠르게 접근할 길이 끊긴다.

- **[M5] 마이크로카피 일부가 길거나 톤 불일치.**
  working 카드 "1~2분 걸릴 수 있어요"는 적절하나, 일부 버블이 길어 모바일에서 부담. 이모지 사용량(🎒🙌🍖☕ 등)이 협업 도구치고 다소 과해 정보 밀도를 떨어뜨림 — 톤(가볍게)과 가독성의 균형 재검토.

### Low

- **[L1] `role="host"`가 메타데이터(영희)와 내 시점 토글 두 곳에 공존.** 데모 토글과 실제 역할이 섞여, "내가 방장인지"가 헷갈릴 수 있다. 실제 구현에선 토글 제거.
- **[L2] 시간 표기 데모 로직(`clock`)이 14:0x로만 증가** — 실데이터 타임스탬프로 교체 필요.
- **[L3] 외부 링크 URL이 한글 쿼리("naver.me/제주-광치기-해변")** — 데모용. 실제 공유 링크 포맷과 다름.
- **[L4] 후보 정렬·필터 없음.** 후보가 많아지면(스크롤 풀) 카테고리 필터/정렬/좋아요순이 필요해질 수 있다.
- **[L5] Tweaks 패널은 디자인 탐색용**이라 제품 빌드에서 제거 대상(명시).

## 5. 추가 검토 필요 항목 (가장 중요)

구현 착수 전에 사용자(제품 오너)와 합의하거나, 디자인을 보강해야 하는 사항.

1. **모바일 디자인을 지금 만들 것인가(결정 필요).** 브리프 최우선 요구 vs 디자이너의 데스크톱 우선 진행이 충돌. 모바일 상단 요약 바·하단 탭(바텀시트)·패널 기본 접힘·단일 컬럼 흐름을 v1에 포함할지, 아니면 브리프의 우선순위를 공식 변경할지 확정해야 한다. 여행 도메인 특성상 모바일 비중이 큰 만큼 가장 먼저 풀어야 할 의사결정.

2. **실데이터 연동 시 깨질 지점(지도·좌표·영업시간).**
   - 지도는 손그림 제주 일러스트(viewBox 0~600 고정 좌표). 실제 Kakao Map JS + 위경도로 교체하면 핀 좌표계·동선 폴리라인·랜드마스가 전부 무효 → 지도 컴포넌트는 재구현 전제. 다른 지역(부산·통영) 방은 제주 일러스트가 그대로 떠 깨진다.
   - Kakao Local은 영업시간 미반환. 카드·상세에 영업시간 필드와 "확인 필요" 단서 슬롯을 미리 설계해야 한다(현재 없음).
   - 거리("성산 0.8km")가 숙소 기준 하드코딩. 실제로는 박별 숙소가 바뀌면 거리·동선이 동적. 정적 텍스트 가정을 깨야 함.
   - 평점("★4.6"), note 설명 등 일부 필드는 출처(Kakao/네이버/Google)별 제공 여부가 달라 누락 시 폴백 표기 필요.

3. **에러·로딩·부분실패 상태 세트 설계.** 링크 파싱 실패, 미지원 도메인, 검색 0건, 일정 생성 실패/타임아웃, WebSocket 재연결, 큐 작업 실패, 권한 만료 등. 특히 일정 생성은 1~2분 장시간 비동기라 중간 실패·부분 결과·재시도 UX가 반드시 필요.

4. **확정 일정 뷰의 분리.** 확정 후에도 작업본은 계속 변한다는 게 핵심 도메인 규칙인데, 현재 일정 탭은 작업본 하나만 보여준다. "확정본 보기 ↔ 작업본 보기" 전환 또는 차이(diff) 표시, 상단 요약 바에서 확정본 펼치기를 어떻게 줄지 결정·디자인 필요.

5. **권한·멤버 엣지케이스.**
   - 방장이 나가거나 권한 위임 시(브리프상 v1 범위 밖이나, 방장 부재 시 확정 불가 데드락은 확인 필요).
   - 동시 편집 충돌: 두 사람이 같은 슬롯을 동시에 수정 요청할 때의 표시(큐잉으로 직렬화되지만 사용자에겐 누구 요청이 처리 중인지 보여야).
   - 비방장 여행자가 본 확정 버튼 비노출은 OK지만, "확정 요청" 같은 방장에게 신호 보내는 경로가 있어야 하는지(현재 안내 문구만).

6. **브리프 열린 질문과의 대응.**
   - 외부 링크 후보 추가는 브리프에 없던 신규 기능(chat1.md에서 추가). 봇 임의 추가 금지 제약과는 어긋나지 않으나, 이 흐름을 v1 공식 범위로 승격할지, 백엔드(링크 메타 파싱·Kakao 매칭) 지원이 가능한지 확인 필요. 임의 블로그/인스타 링크에서 좌표·카테고리를 신뢰성 있게 뽑는 건 난도가 높다.
   - 후보 상세 팝업의 카카오/네이버/구글 외부 링크는 단순 검색 쿼리 URL이라 정확도가 낮음. place_id 기반 딥링크가 가능한지(Kakao는 가능, 네이버·구글은 제약) 확인.

7. **선호의 누적·반영 가시화.** "좋아요/싫어요"는 있으나, 자유발화 선호("난 카페 좋아")의 수집·표시, 그리고 그 선호가 다음 추천에 반영됐다는 피드백(예: 추천 카드의 "○○님 선호 반영")이 없다. 선호가 보이지 않으면 기능 가치가 약해진다.

8. **초대 링크 생성·공유 UI.** 입장(JoinRoom)은 있으나, 방장이 초대 링크를 만들고 복사·공유하는 화면이 없다. v1 최소 진입에 필요한지 확인(브리프는 "초대 링크 입장"만 명시, 생성은 모호).
