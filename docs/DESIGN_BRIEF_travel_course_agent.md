# 디자인 브리프 — 여행 코스 에이전트 (그룹 협업 웹앱)

> **디자인 에이전트에게:** 당신은 아래 제품의 UX/UI를 디자인합니다. 이 문서는 자체 완결형이며, 시각 디자인(색·타이포·간격·모션·아이콘)은 당신이 정합니다. 단, "구조·흐름·컴포넌트·제약"은 이미 확정되어 있으니 **그대로 따르고, '비범위(맨 아래)'의 확정 제약은 바꾸지 마세요.**
> 산출물: 데스크톱/모바일 양쪽의 주요 화면, 4종 카드 컴포넌트, 상태별(작업 중·큐잉·빈 상태·되물음·권한 막힘) 화면. 모든 문구는 **한국어**.

---

## 0. 제품 한 줄 요약

여러 여행자가 한 **방**에 모여, **봇**을 명시적으로 호출해 장소를 검색·등록하고, 봇이 **숙소**와 **동선**을 고려해 짜 주는 **일정**을 함께 다듬다가, **방장**이 확정하는 그룹 협업 웹앱. 자체 웹앱이며 카카오톡 같은 외부 메신저가 아니다. 국내 여행(제주 등)이 1차 맥락.

---

## 1. 도메인 용어 (이 단어들로 화면을 만든다)

- **방**: 여행자들과 봇이 모여 하나의 여행을 논의·설계하는 공간.
- **여행자**: 방에 참여한 사람 멤버.
- **방장**: 방을 만든 여행자. 일정을 확정할 수 있는 유일한 역할.
- **봇**: 에이전트. 호출 시 응답하며 카드를 만든다.
- **후보 장소**: 여행자가 검색해 방에 담아둔 방문 후보(명소·식당 등). 방 전체가 공유하는 풀. 아직 일정에 들어가지 않은 재료.
- **일정**: 후보 장소 중 일부를 골라 숙소와 동선을 고려해 날짜·시각순으로 배치한 결과물. **방마다 단 하나의 "작업 중 일정"** 만 존재하며 반복 수정된다(경쟁하는 여러 안을 동시에 두지 않는다).
- **확정 일정**: 방장이 확정한 시점의 일정을 복제한 스냅샷. 작업 중 일정은 계속 바뀌어도 확정 일정은 다시 확정하기 전까지 고정.
- **동선**: 일정 안에서 숙소·장소 사이의 이동(거리·소요시간·교통수단). 별도 화면이 아니라 일정 안에 녹인다.
- **숙소**: 동선 계산의 기준점이 되는 박별 거점. 박마다 다를 수 있다(기본은 전 일정 동일).
- **선호**: 개별 여행자의 취향 신호(좋아요/싫어요, "난 카페 좋아" 같은 발화). 다음 추천에 반영.

## 2. 지켜야 할 UX 원칙 (제약)

1. **봇은 명시적 호출에만 응답한다.** `@봇` 멘션 또는 슬래시 커맨드(`/일정`, `/추천` 등)일 때만. 사람끼리 대화에 끼어들지 않는다 → 입력창에 "봇 부르기" 수단이 항상 분명해야 한다.
2. **봇의 답은 채팅 버블이 아니라 카드다.** 장소 옵션·일정·지도는 구조화된 카드로. 짧은 대화성 답변만 버블.
3. **지속 상태는 패널에 산다.** 후보 풀·일정·지도는 채팅 스크롤에 묻히지 않게 사이드 패널에서 항상 접근 가능.
4. **확정은 방장만.** 확정 액션은 방장에게만 노출(일반 여행자에겐 비노출).
5. **긴 작업은 막지 않는다.** 일정 설계는 1~2분 걸릴 수 있다. 그동안 사람은 계속 대화 가능, 봇은 "작업 중"을 보여 주고, 새 호출은 큐에 쌓인다.
6. **봇은 방을 안다.** 단순 질의에도 방 맥락(목적지·날짜·확정 일정)을 반영해 답하고, 이미 아는 것을 되묻지 않는다.
7. **한국어 UI.**

## 4. 레이아웃 — 채팅 + 접이식 사이드 패널 (반응형)

데스크톱은 채팅과 사이드 패널을 좌우로, 모바일은 채팅을 전면에 두고 패널을 하단 탭(바텀시트)으로 접는다.

```
[ 데스크톱 ]
+----------------------+---------------------------+
|  채팅                 |  사이드 패널               |
|  [민수] 우도 갈까?     |  ( 후보 | 일정 | 지도 )    |  <- 탭
|  [영희] 좋아           |  +---------------------+  |
|  @봇 일정 짜줘         |  |  (선택된 탭 내용)     |  |
|  +----------------+   |  |                     |  |
|  |  일정 카드      |   |  |                     |  |
|  +----------------+   |  +---------------------+  |
|  [@봇] [입력...]   [>] |  (패널 접기/펼치기 가능)   |
+----------------------+---------------------------+

[ 모바일 ]
+----------------------+
| 제주 3박 4일      v   |  <- 상단 요약 바(확정 일정 핵심)
+----------------------+
|  [민수] 우도 갈까?     |
|  @봇 일정 짜줘         |
|  +----------------+   |
|  |  일정 카드      |   |
|  +----------------+   |
+----------------------+
| [후보] [일정] [지도]   |  <- 하단 탭(누르면 바텀시트)
| [@봇] [입력...]        |
+----------------------+
```

- **상단 요약 바**(모바일 필수, 데스크톱 선택): 현재 방의 핵심 — 목적지·기간·확정 일정 여부. 누르면 확정 일정 펼침.
- **패널 접힘 기본값**: 데스크톱은 펼친 상태, 모바일은 접힌 상태(탭으로 호출).
- **방 진입(별도 최소 화면)**: 위 레이아웃은 방 *안*의 화면이다. 방 밖에는 "내 방 목록 + 방 만들기 + 초대 링크 입장"만 두는 최소 진입 화면을 둔다(고급 방 관리는 보류).

## 5. 영역별 명세

### 5.1 채팅 영역
- **메시지 목록**: 발화자 이름 태그(`[민수]`). 봇 메시지는 사람과 시각적으로 구분.
- **봇 호출 수단**: 입력창 좌측에 `@봇` 부르기 버튼/칩(누르면 멘션 삽입 + "봇에게 보냄" 상태). `/` 입력 시 슬래시 커맨드 자동완성.
- **슬래시 커맨드(최소 세트)**: `/일정`(생성·수정), `/추천`(장소 추천), `/검색 <키워드>`(옵션 카드), `/확정`(방장 전용).
- 봇 응답 카드는 채팅에 인라인으로 뜨고, 동시에 관련 패널 탭이 갱신된다.

### 5.2 사이드 패널 — 탭 3종
- **후보 탭(후보 장소 풀)**: 방 공유 후보 목록. 항목마다 이름·카테고리·좋아요/싫어요·삭제. 비어 있으면 빈 상태 안내.
- **일정 탭(작업 중 일정)**: 단일 작업 중 일정의 타임라인. 박별 **숙소** 슬롯(검색 흐름에서 "숙소로 지정"). 방장에게 **확정** 버튼 노출. 확정 일정이 있으면 "확정됨" 표시 + 작업본과의 관계 명시.
- **지도 탭**: 지도(국내 지도 SDK). 후보·일정 장소·숙소를 핀으로, 동선을 선으로. 일정 항목 선택 시 지도 포커스.

## 6. 카드 컴포넌트 카탈로그 (4종)

봇이 만드는 카드. 각 카드의 *내용과 액션*을 디자인한다.

| 카드 | 핵심 내용 | 액션 |
|---|---|---|
| **장소 옵션** | 검색 결과 N개(이름·카테고리·거리·간단 정보) | 각 항목 **+추가**(후보 풀로) / **숙소로 지정** / 지도 보기 |
| **일정 타임라인** | 날짜·시각순 장소 배치, 이동(동선) 표시, 박별 숙소 | 항목 수정 요청 / (방장) **확정** |
| **옵션 비교** | 한 항목(슬롯)의 대안 2~3개(예: 점심 식당 후보) | 하나 선택해 일정에 반영 |
| **지도** | 핀(장소·숙소) + 동선 선 | 핀 탭 → 상세 / 후보 추가 |

- **후보 등록의 유일한 경로는 장소 옵션 카드의 "추가"** 다(봇이 임의 추가하지 않음).
- **동선 표기**: 일정 카드의 장소 사이에 "이동 N분(교통수단)"을 보여 준다.
- **옵션 비교는 사용자 요청 시에만** 뜬다(슬롯 단위 대안). 봇이 자동으로 여러 안을 제시하지 않는다.

## 7. 핵심 사용자 흐름

**A. 장소 검색 → 후보 추가**
1) 민수: `@봇 성산 근처 흑돼지`(또는 `/검색 성산 흑돼지`)
2) 봇: **장소 옵션 카드**(후보 3~5개)
3) 민수: 마음에 드는 항목 **+추가** → **후보 탭**에 등록, 모두에게 공유

**B. 일정 생성 (작업 경로)**
1) 영희: `@봇 후보로 2일 일정 짜줘`
2) 봇: "작업 중" 표시(단계: 일정 설계 → 동선 → 검증). 사람들은 그동안 자유롭게 대화 가능
3) 봇: **일정 타임라인 카드** + **일정 탭**·**지도 탭** 갱신

**C. 일정 수정 / 모호할 때 되물음**
1) 민수: `@봇 우도를 둘째 날로 옮겨줘` → 봇이 작업 중 일정을 수정해 새 카드
2) 맥락이 모호하면 봇이 **되물음** 메시지로 한 번 확인

**D. 방장 확정**
1) 일정 탭/일정 카드의 **확정** 버튼은 방장에게만 보임
2) 방장 확정 → **확정 일정** 스냅샷 생성, 상단 요약 바·일정 탭에 "확정됨" 표시
3) 이후 작업 중 일정은 바뀌어도 확정 일정은 다시 확정 전까지 고정

**E. 단순 질의 (즉답)**
1) 민수: `@봇 내일 비 와?`
2) 봇: 방 맥락(제주·날짜)을 반영해 **즉답**(버블). 맥락으로도 모호할 때만 되물음

**F. 선호 표현**
1) 여행자가 후보 카드/탭에서 **좋아요/싫어요** → 개인 **선호**로 누적(자유발화도 수집)
2) 좋아요는 **방 전체에 공유**(누가 눌렀는지 표시), 싫어요는 집계 위주로 부드럽게

## 8. 상태와 피드백

- **작업 중**: "작업 중" 상태를 명확히 표시. 가능하면 단계(설계/동선/검증) 힌트. **취소 버튼**을 함께 둔다(진행 작업 폐기).
- **큐잉**: 작업 중 새 `@봇` 호출이 오면 "대기 중(앞 작업 후 처리)" 알림.
- **되물음**: 봇의 명확화 질문은 일반 메시지로, 답하기 쉽게(빠른 선택지 칩 권장).
- **빈 상태**: 새 방("여행지를 정하고 장소를 검색해 담아보세요" + `@봇` 예시) / 후보 비어 있음 / 일정 없음 / 확정 전 — 각각 안내.
- **권한 막힘**: 일반 여행자가 확정을 시도할 경로 자체를 숨김(에러 대신 비노출).

## 10. 확정된 세부 사항

- **작업 취소**: "작업 중" 옆 **취소 버튼**. 선점(새 요청이 자동 취소)은 하지 않음 — 취소는 명시적 버튼으로만.
- **옵션 비교**: 사용자 요청 시에만, 한 슬롯의 대안 2~3개로 한정. 자동 제시 없음.
- **선호 가시성**: 방 전체 공유(좋아요는 누가 눌렀는지, 싫어요는 집계 위주).
- **숙소 지정**: 장소 검색 흐름 재사용 + "숙소로 지정"(후보 풀이 아니라 일정의 박별 숙소 슬롯). 별도 검색 UI 없음.
- **방 목록/전환**: 최소 진입(내 방 목록 + 방 만들기 + 초대 링크 입장)만 v1 포함. 본격 관리 보류. 3분할 레이아웃은 쓰지 않음.

## 11. 비주얼·디자인 시스템 힌트

- **반응형 우선순위**: 여행 도메인 특성상 모바일 비중이 큼 → 모바일 가독성 1순위, 데스크톱은 패널 활용.
- **지도**: 국내 지도 SDK(핀·동선 선 스타일은 디자인이 정의).
- **톤**: 협업·여행 — 가볍고 명료하게.
- (참고 패턴이 있으면 카드/타임라인/지도 레이아웃에 재사용하되, 색·타이포는 새로 정의.)

## 12. 비범위 (건드리지 말 것)

- **봇 트리거 정책**(명시적 호출만), **확정 권한**(방장 전용), **단일 작업 일정 모델**(경쟁 안 없음), **후보 등록 경로**(옵션 카드의 "추가") 같은 확정 제약은 바꾸지 말 것.
- 백엔드·에이전트 구조, 데이터 저장, API 선택은 디자인 범위 밖.
- 결제·과금·정식 권한 관리(멤버 추방 등)는 v1 범위 밖.
