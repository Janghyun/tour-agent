# 여행 코스 에이전트 — UX 명세 (디자인 핸드오프)

> 목적: 디자인 에이전트(예: Claude Design)에게 넘길 UX 구조·흐름·컴포넌트·제약 정의서.
> **이 문서는 시각 디자인(색·타이포·간격·모션)을 정하지 않는다.** 그것은 디자인 단계의 몫이다.
> 여기서는 "무엇이 화면에 있고, 어떻게 흐르고, 무엇을 지켜야 하는가"만 못 박는다.
> 도메인 용어는 루트 `CONTEXT.md`, 기능·기술 결정은 `docs/TASK_travel_course_agent.md`(특히 2.5절) 참조.

---

## 1. 한 줄 요약

여러 여행자가 한 **방**에 모여, **봇**을 명시적으로 호출해 장소를 검색·등록하고, 봇이 **숙소**와 **동선**을 고려해 짜 주는 **일정**을 함께 다듬다가, **방장**이 확정하는 그룹 협업 웹앱.

---

## 2. 지켜야 할 UX 원칙 (제약)

1. **봇은 명시적 호출에만 응답한다.** `@봇` 멘션 또는 슬래시 커맨드(`/일정`, `/추천` 등)일 때만. 사람끼리의 대화에 끼어들지 않는다. → 입력창에 "봇 부르기" 수단이 항상 분명해야 한다.
2. **봇의 답은 채팅 버블이 아니라 카드다.** 장소 옵션·일정·지도는 구조화된 카드로 표시한다(자유 텍스트 파싱 금지). 짧은 대화성 답변만 버블로.
3. **지속 상태는 패널에 산다.** 후보 풀·일정·지도는 채팅 스크롤에 묻히지 않게 사이드 패널에서 항상 접근 가능해야 한다.
4. **확정은 방장만.** 확정 액션은 방장에게만 노출한다. 일반 여행자에게는 보이지 않거나 비활성.
5. **긴 작업은 막지 않는다.** 일정 설계는 1~2분 걸릴 수 있다. 그동안 사람은 계속 대화할 수 있고, 봇은 "작업 중"을 보여 주며, 새 호출은 큐에 쌓인다.
6. **봇은 방을 안다.** 단순 질의에도 방 맥락(목적지·날짜·확정 일정)을 반영해 답한다. 이미 아는 것을 되묻지 않는다.
7. **한국어 UI.** 모든 문구는 한국어. 국내 여행(제주 등)이 1차 사용 맥락.

## 3. 사용자와 역할

| 역할 | 설명 | 권한 |
|---|---|---|
| **여행자** | 방에 참여한 사람 멤버 | 메시지, 봇 호출, 후보 장소 추가/삭제, 선호 표시(좋아요/싫어요), 일정 의견 |
| **방장** | 방을 만든 여행자 | 여행자 권한 전부 + **일정 확정**, 멤버 관리(추후) |
| **봇** | 에이전트 | 호출 시 응답. 카드 생성. 임의로 후보를 추가하거나 일정을 확정하지 않음 |

## 4. 레이아웃 — 채팅 + 접이식 사이드 패널 (반응형)

데스크톱은 채팅과 사이드 패널을 좌우로, 모바일은 채팅을 전면에 두고 패널을 하단 탭(바텀시트)으로 접는다.

```
[ 데스크톱 ]
+----------------------+---------------------------+
|  채팅                 |  사이드 패널               |
|  [민수] 우도 갈까?     |  ( 후보 | 일정 | 지도 )    |  <- 탭
|  [영희] 좋아           |  +---------------------+  |
|  @봇 일정 짜줘         |  |  (선택된 탭 내용)     |  |
|  +----------------+   |  |                     |  |
|  |  일정 카드      |   |  |                     |  |
|  +----------------+   |  +---------------------+  |
|  [@봇] [입력...]   [>] |  (패널 접기/펼치기 가능)   |
+----------------------+---------------------------+

[ 모바일 ]
+----------------------+
| 제주 3박 4일      v   |  <- 상단 요약 바(확정 일정 핵심)
+----------------------+
|  [민수] 우도 갈까?     |
|  @봇 일정 짜줘         |
|  +----------------+   |
|  |  일정 카드      |   |
|  +----------------+   |
+----------------------+
| [후보] [일정] [지도]   |  <- 하단 탭(누르면 바텀시트)
| [@봇] [입력...]        |
+----------------------+
```

- **상단 요약 바**(모바일 필수, 데스크톱 선택): 현재 방의 핵심 — 목적지·기간·확정 일정 여부. 누르면 확정 일정 펼침.
- **패널 접힘 기본값**: 데스크톱은 펼친 상태, 모바일은 접힌 상태(탭으로 호출).
- **방 진입(별도 최소 화면)**: 위 레이아웃은 방 *안*의 화면이다. 방 밖에는 "내 방 목록 + 방 만들기 + 초대 링크 입장"만 두는 최소 진입 화면을 둔다(고급 방 관리는 보류).

## 5. 영역별 명세

### 5.1 채팅 영역

- **메시지 목록**: 발화자 이름 태그 표시(`[민수]`). 봇 메시지는 사람과 시각적으로 구분.
- **봇 호출 수단**(원칙 1):
  - 입력창 좌측에 `@봇` 부르기 버튼(또는 칩). 누르면 멘션이 입력에 삽입되고 "봇에게 보냄" 상태가 됨.
  - `/` 입력 시 슬래시 커맨드 자동완성 표시.
- **슬래시 커맨드(최소 세트, 디자인 단계에서 확장 가능)**:
  - `/일정` — 일정 생성·수정 요청
  - `/추천` — 장소 추천 요청
  - `/검색 <키워드>` — 장소 검색 → 옵션 카드
  - `/확정` — (방장 전용) 현재 작업 중 일정을 확정
- **봇 응답 카드**는 채팅 흐름 안에 인라인으로 뜨고, 동시에 관련 패널 탭이 갱신된다(예: 일정 카드 → 일정 탭 업데이트).

### 5.2 사이드 패널 — 탭 3종 (+숙소)

- **후보 탭 (후보 장소 풀)**: 방이 공유하는 후보 목록. 항목마다 이름·카테고리·좋아요/싫어요·삭제. 비어 있으면 빈 상태 안내(아래 8절).
- **일정 탭 (작업 중 일정)**: 단일 작업 중 일정의 타임라인. 박별 **숙소** 슬롯(장소 검색 흐름에서 "숙소로 지정"). 방장에게 **확정** 버튼 노출. 확정 일정이 있으면 "확정됨" 표시 + 작업본과의 관계 명시.
- **지도 탭**: Kakao Map. 후보·일정 장소·숙소를 핀으로, 동선을 선으로. 일정 탭과 연동(일정 항목 선택 시 지도 포커스).

## 6. 카드 컴포넌트 카탈로그

봇은 UI 툴(`present_*`)을 호출해 카드를 만든다. 디자인 에이전트는 아래 각 카드의 *내용과 액션*을 디자인한다.

| 카드 | 백엔드 툴 | 핵심 내용 | 액션 |
|---|---|---|---|
| **장소 옵션** | `present_place_options` | 검색 결과 N개(이름·카테고리·거리·간단 정보) | 각 항목 **+추가**(후보 풀로), 지도 보기 |
| **일정 타임라인** | `present_itinerary` | 날짜·시각순 장소 배치, 이동(동선) 표시, 박별 숙소 | 항목 수정 요청, (방장) **확정** |
| **옵션 비교** | (슬롯 대안 비교) | 한 항목의 대안 2~3개(예: 점심 식당 후보) | 하나 선택해 일정에 반영 |
| **지도** | `present_map` | 핀(장소·숙소) + 동선 선 | 핀 탭 → 상세, 후보 추가 |

- **장소 옵션 카드의 "추가"가 후보 등록의 유일한 경로**다(봇이 임의 추가하지 않음 — 원칙 1과 일관).
- **일정 카드의 동선 표기**: 장소 사이에 "이동 N분(교통수단)"을 보여 준다. 동선은 별도 화면이 아니라 일정 안에 녹인다.
- **옵션 비교는 사용자 요청 시에만** 뜬다(슬롯 단위 대안). 봇이 자동으로 여러 안을 제시하지 않는다 — 단일 작업 일정 모델과 일관.

## 7. 핵심 사용자 흐름

### A. 장소 검색 → 후보 추가
1. 민수: `@봇 성산 근처 흑돼지` (또는 `/검색 성산 흑돼지`)
2. 봇: **장소 옵션 카드**(후보 3~5개)
3. 민수: 마음에 드는 항목 **+추가** → **후보 탭**에 등록, 모두에게 공유

### B. 일정 생성 (작업 경로)
1. 영희: `@봇 후보로 2일 일정 짜줘`
2. 봇: "작업 중" 표시(서브에이전트 단계: 일정 설계 → 동선 → 검증)
3. 사람들은 그동안 자유롭게 대화 가능
4. 봇: **일정 타임라인 카드** + **일정 탭**·**지도 탭** 갱신

### C. 일정 수정 / 모호할 때 되물음
1. 민수: `@봇 우도를 둘째 날로 옮겨줘` → 봇이 작업 중 일정을 수정해 새 카드
2. 맥락이 모호하면(예: 어떤 우도 일정인지 불명확) 봇이 **되물음** 메시지로 한 번 확인

### D. 방장 확정
1. 일정 탭(또는 일정 카드)의 **확정** 버튼은 방장에게만 보임
2. 방장이 확정 → 현재 작업본을 복제한 **확정 일정** 스냅샷 생성, 상단 요약 바·일정 탭에 "확정됨" 표시
3. 이후 작업 중 일정은 계속 바뀔 수 있으나, 확정 일정은 방장이 다시 확정하기 전까지 고정

### E. 단순 질의 (치프 경로)
1. 민수: `@봇 내일 비 와?`
2. 봇: 방 맥락(제주·날짜)을 반영해 **즉답**(버블). 맥락으로도 모호할 때만 되물음

### F. 선호 표현
1. 여행자가 후보 카드/탭에서 **좋아요/싫어요** 표시 → 개인 **선호**로 누적
2. 다음 추천·일정 설계에 반영(자유발화 "난 카페 좋아"도 선호로 수집)
3. 좋아요는 **방 전체에 공유**(누가 눌렀는지 표시), 싫어요는 집계 위주로 부드럽게

## 8. 상태와 피드백

- **작업 중**: 봇이 "작업 중" 상태를 명확히 표시. 가능하면 단계(설계/동선/검증) 진행 힌트. **취소 버튼**을 함께 둔다(진행 작업 폐기 + 락 해제).
- **큐잉**: 작업 중에 새 `@봇` 호출이 오면 "대기 중(앞 작업 후 처리)"임을 알림.
- **되물음**: 봇의 명확화 질문은 일반 메시지로, 답하기 쉽게(빠른 선택지 칩 권장).
- **빈 상태**:
  - 새 방: "여행지를 정하고 장소를 검색해 담아보세요" + `@봇` 사용 예시.
  - 후보 비어 있음 / 일정 없음 / 확정 전 — 각각 안내 문구.
- **권한 막힘**: 일반 여행자가 확정을 시도할 경로 자체를 숨김(에러 대신 비노출).

## 12. 결정된 세부 사항 (2026-06-03)

- **작업 취소**: "작업 중" 옆에 **취소 버튼**을 둔다. 누르면 진행 작업 폐기 + 세션 락 해제(쓴 토큰은 매몰). 선점(새 요청이 자동 취소)은 하지 않는다 — 취소는 명시적 버튼으로만.
- **옵션 비교**: **사용자 요청 시에만**, 그리고 **한 항목(슬롯)의 대안 2~3개** 비교로 한정한다(예: "점심 다른 데 없어?"). 봇이 자동으로 여러 안을 제시하지 않는다 — 단일 작업 일정 모델과 일관.
- **선호 가시성**: **방 전체에 공유**한다(누가 무엇을 좋아하는지 보임). 단 "싫어요"는 사회적으로 민감하므로 **집계 위주**로 부드럽게, 좋아요는 누가 눌렀는지 공유.
- **숙소 지정**: 장소 **검색 흐름을 재사용**한다. 같은 검색 카드에서 "추가" 대신 **"숙소로 지정"** 을 선택하면 후보 풀이 아니라 일정의 박별 숙소 슬롯으로 들어간다. 별도 검색 UI를 만들지 않는다.
- **방 목록/전환**: 최소 진입(**내 방 목록 + 방 만들기 + 초대 링크 입장**)만 v1에 포함. 아카이브·검색·정렬 등 본격 관리는 보류. 방 전환은 상단 드롭다운 또는 별도 목록 화면(3분할 레이아웃은 쓰지 않는다).
